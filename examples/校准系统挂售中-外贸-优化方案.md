# 「校准系统挂售中-外贸」优化方案

## 一、现有流程概述

**业务目的**：定时校准外贸挂售商品是否仍在售，通过外部 API 查询尺码列表来验证，有结果则更新校准时间，无结果则删除挂售记录。

**执行链路**：
```
定时触发 → 取100条外贸挂售记录(按校准时间升序) → 逐条进子流程:
  ├─ 每条都查一次芝麻代理配置
  ├─ 每条都查一次比价数据(取snk_id)
  ├─ 条件判断(snk_id是否存在)
  │   ├─ 存在 → 并行调鞋子API + 衣服API
  │   │        ├─ 成功且有数据 → 更新jzsj=当前时间
  │   │        └─ 成功但无数据 → 删除挂售记录
  │   └─ 不存在 → 通知张磊、李义、李聪
  └─ 结束
```

---

## 二、问题诊断

### 问题1：代理配置重复查询（高优先级）

**现状**：每条记录进入子流程都会查一次「芝麻代理」表（筛选：国家=国外 AND snk价格请求状态=正常）。100条记录 = 查100次，但每次返回的都是同一条代理记录。

**影响**：100次无意义的数据库查询，增加执行时间 ~10-20秒。

**优化**：将代理查询上移到主流程，在查询工作表之后、调用子流程之前获取一次，通过子流程参数传入 IP 和 port。

### 问题2：鞋子和衣服 API 同时调用（高优先级）

**现状**：每条记录同时并行调用「鞋子尺码API」和「衣服尺码API」，但一个商品要么是鞋子要么是衣服，不会两者都是。

**影响**：
- 每条记录多发一次完全无用的外部 API 请求
- 浪费芝麻代理流量（IP+port 都传了）
- 100条记录 = 浪费100次外部API调用

**优化**：在挂售表或比价数据中应该有品类字段。根据品类判断调用哪个 API：
```
条件分支(品类):
  ├─ 鞋子 → 只调鞋子API
  └─ 衣服 → 只调衣服API
```

如果数据中没有品类字段，可以：
- 方案A：在挂售表-新逻辑中增加「品类」字段
- 方案B：从比价数据中取品类信息
- 方案C（保守）：先调鞋子API，失败后再调衣服API（串行降级）

### 问题3：删除操作过于激进（中优先级）

**现状**：外部API返回数据条数=0时，直接删除挂售记录。

**风险**：
- 外部API临时故障 → 误删正常的挂售记录
- 商品临时下架后重新上架 → 挂售记录已被永久删除
- 无法追溯什么时候、为什么被删除

**优化**：
- 方案A：改为「标记下架」而非删除（增加状态字段：挂售中/已下架/待确认）
- 方案B：删除前检查连续校准失败次数，连续3次失败才删除
- 方案C：至少在删除前记录日志（写入操作日志表）

### 问题4：无失败重试机制（中优先级）

**现状**：外部API调用（鞋子/衣服尺码查询）如果 status != "success"，走到分支2「无后续节点」，直接跳过。

**影响**：临时网络波动 → 该记录本轮不被校准，要等下一轮定时触发。

**优化**：
- 增加重试1次的机制
- 或者在 status != "success" 时记录失败原因到挂售表的某个字段，下次优先处理

### 问题5：批次量固定100（低优先级）

**现状**：每次定时触发只处理100条，按校准时间升序。

**评估**：
- 如果外贸挂售记录总量 >1000，100条/批的覆盖速度可能不够
- 但如果触发频率够高（如每5分钟），100条是合理的
- 需要确认触发间隔和总记录量来评估

### 问题6：子流程命名不规范（低优先级）

**现状**：子流程名为「未命名子流程」。

**影响**：维护困难，其他人无法从名称理解用途。

**优化**：重命名为「校准单条挂售记录-外贸」。

## 三、优化方案总览

| # | 优化项 | 优先级 | 预期收益 | 实施难度 |
|---|--------|--------|----------|----------|
| 1 | 代理查询上移到主流程 | 高 | 减少100次重复查询 | 低 |
| 2 | 按品类分流鞋子/衣服API | 高 | 减少50%外部API调用 | 中 |
| 3 | 删除改为标记下架 | 中 | 避免误删，可追溯 | 中 |
| 4 | 增加外部API失败重试 | 中 | 提高校准成功率 | 低 |
| 5 | 评估并调整批次量 | 低 | 加快全量覆盖速度 | 低 |
| 6 | 子流程重命名 | 低 | 提高可维护性 | 最低 |

## 四、优化后目标架构

```
定时触发
  │
  ├─ [获取单条] 查询芝麻代理 (只查1次)
  │   筛选: 国家=国外 AND snk价格请求状态=正常
  │
  ├─ [获取多条] 查询挂售表-新逻辑
  │   筛选: 业务类型=外贸
  │   排序: 挂售校准时间 升序
  │   限制: 100条
  │
  └─ [子流程] 校准单条挂售记录-外贸
      │  参数: 挂售记录 + 代理IP + 代理port
      │
      ├─ [获取单条] 查询比价数据
      │   筛选: global_sku_id = 当前记录.global_sku_id
      │
      ├─ [条件分支] snk_id 是否存在？
      │   │
      │   ├─ 不存在 → 通知告警
      │   │
      │   └─ 存在 → [条件分支] 品类判断
      │       │
      │       ├─ 鞋子 → 调鞋子API
      │       │   ├─ success且有数据 → 更新jzsj
      │       │   ├─ success但无数据 → 标记「待确认下架」
      │       │   └─ 失败 → 重试1次 / 记录失败
      │       │
      │       └─ 衣服 → 调衣服API
      │           ├─ success且有数据 → 更新jzsj
      │           ├─ success但无数据 → 标记「待确认下架」
      │           └─ 失败 → 重试1次 / 记录失败
      │
      └─ 结束
```

## 五、实施步骤

### 第一步：代理查询上移（最简单，立即可做）

1. 在主流程「查询工作表」节点之前，增加「获取单条：芝麻代理」节点
2. 子流程增加两个输入参数：`proxy_ip`、`proxy_port`
3. 子流程调用节点传入代理 IP 和 port
4. 删除子流程内部的「代理」查询节点
5. 鞋子/衣服的字段映射改为从输入参数取 IP/port

### 第二步：按品类分流（需确认数据字段）

1. 确认挂售表或比价数据中是否有「品类」字段（鞋子/衣服/配饰等）
2. 在子流程中，snk_id存在之后、并行分支之前，增加品类条件分支
3. 将现有并行分支拆为两条互斥路径
4. 删除不需要的并行结构

### 第三步：删除改标记（需加字段）

1. 在挂售表-新逻辑中增加「校准状态」字段（选项：正常/待确认下架/已下架）
2. 将现有的「删除记录」节点改为「更新记录」，设置校准状态=待确认下架
3. 另建定时工作流：定期检查「待确认下架」记录，连续N次仍为待确认则改为「已下架」
4. 主流程筛选条件增加：校准状态 != 已下架
